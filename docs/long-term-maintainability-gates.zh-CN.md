# WR.Next 长期维护工程指标

## 结论

后续开发必须同时满足两类目标：

1. 功能逐段打通
2. 工程形态持续可维护

如果只能得到“暂时能跑”，但会继续制造路径散乱、职责混杂、反射蔓延、验证不可复现的问题，就不算可接受推进。

## 一、总原则

后续所有窄包都按下面四个判断标准验收：

1. 是否落在既定职责层
2. 是否可重复验证
3. 是否会扩大后续返工面
4. 是否会把临时桥接固化成长期债务

只满足第 2 条、不满足其余三条的改动，不进主线。

## 二、源码维护指标

### 1. 分层归属必须明确

新增代码只能优先落在既有分层中：

- `App`
- `Shell`
- `Views`
- `ViewModels`
- `Adapters`
- `Runtime`
- `Models`
- `Infrastructure`
- `Diagnostics`

判断标准只有一句话：

> 新文件是否能一眼看出职责归属。

### 2. 页面层不得继续吞运行时逻辑

`xaml.cs` / 页面 code-behind 只允许保留：

- 初始化
- 事件转发
- 最小装配

禁止继续把下面这些直接塞进页面层：

- 产品启动主链
- 运行时状态机
- 路径拼装
- 配置持久化
- 反射装载细节

### 3. 反射只能留在边界

反射不是默认手段，只能作为宿主与动态模块之间的边界适配：

- 产品 DLL
- 插件 DLL
- 自定义战斗脚本 DLL
- 原版未公开但必须桥接的设置对象

禁止把反射继续扩散为：

- 页面取值常规方式
- 核心业务调度方式
- 跨层通信默认方式

长期目标是：

> 静态引用承载基础底座，反射只承载动态模块边界。

### 4. 运行时路径必须集中

所有运行目录、模块目录、日志目录、数据目录、配置目录，都必须由统一路径入口给出。

禁止新增：

- 页面里硬编码绝对路径
- 多处重复拼 `Bin` / `Data` / `Profiles`
- 同一种运行物出现多个查找入口

## 三、发布维护指标

### 1. 运行目录必须可读

运行根必须让人一眼区分：

- 主程序
- `Bin` 依赖
- `Data` 数据
- `Products/Plugins/FightClass` 模块
- `Logs` 日志

后续不再接受“根目录散一地 DLL / 探针 / 临时文件”。

### 2. 发布脚本必须是唯一真源

运行布局不允许靠手工复制维持。

判断标准：

> 任何一次可交付产物，都必须能由发布脚本重新产出。

这条尤其重要，因为后面会越来越依赖：

- 原版运行时依赖
- 第三方库
- 数据目录
- 动态模块目录

### 3. 临时诊断物必须隔离

探针、审计日志、实验工具可以保留，但必须隔离在：

- `Diagnostics`
- `tools`
- `docs`

不得混入正式运行主链目录和页面逻辑。

## 四、功能推进指标

### 1. 必须按能力层推进

后续主线按五层能力图推进：

1. 对象层
2. 状态层
3. 导航层
4. 战斗脚本层
5. 施法执行层

禁止再回到“看哪个页面顺眼就补哪个页面”的推进方式。

### 2. 每次只压一个明确断点

每个窄包都必须说明：

- 当前卡在哪一层
- 当前验证了哪一个断点
- 当前没有继续扩到哪里

这样做的目的不是保守，而是防止再次出现：

- UI 还没稳就乱接内核
- 进程链没闭合就硬补战斗页
- 设置问题和执行问题混在一起

### 3. 先保链完整，再谈体验细化

对当前项目，优先级必须固定为：

1. 主链完整
2. 主链稳定
3. 主链可观测
4. 页面细节优化

也就是说，后续只要战斗主链还没打穿，就不能让 UI 小修补继续吞掉主要开发带宽。

## 五、验证维护指标

### 1. 每个结论都要可复查

后续文档必须优先回答三件事：

1. 结论是什么
2. 证据是什么
3. 下一步只做什么

避免再次堆大量噪音日志却没有硬结论。

### 2. 验证口径必须稳定

对于主链推进，优先保留这三类稳定证据：

- 代码入口位置
- 运行日志关键点
- 运行产物目录/依赖状态

尽量少依赖：

- 单次手点现象
- 不可复现的截图理解
- 没有上下文的零散异常

### 3. 不把“能启动”误判为“可接管”

后续所有结论要分清：

- `can launch`
- `can load`
- `can start`
- `can act`
- `can sustain`

只有一路向后打通，才算真实推进。

## 六、当前阶段的硬约束

结合当前项目状态，最近阶段只允许做两类事情：

1. 沿 `WRotation` 主链继续下钻
2. 为后续接管降低工程债务

最近阶段不应再扩散到：

- 大范围 UI 复刻
- 多产品并行接线
- 无边界的新探针体系
- 没有主链收益的页面填充

## 七、一句话门槛

以后判断一项工作值不值得进主线，只问两句：

1. 它有没有让主链更完整、更稳定、更可验证？
2. 它有没有让后续维护更简单，而不是更乱？

两句里只要有一句答不上来，这项改动就先别进。

## 八、下一步建议

按这份指标，下一包最合理的方向仍然是：

1. 只盯 `WRotation` 战斗执行层
2. 把“已进入战斗决策但未形成真实动作”的断点钉死
3. 用结构化证据区分到底卡在：
   - 目标状态
   - 跟近/移动
   - 自定义脚本调度
   - spell/lua 执行

这才是当前最值得花时间的位置。
